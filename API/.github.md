Necesito tu ayuda para programar un sistema backend. Aquí tienes el contexto total del proyecto, la arquitectura y las reglas de negocio. Debes apegarte estrictamente a estos lineamientos en todas tus respuestas.

1. Contexto del Proyecto
Estamos desarrollando una API RESTful para gestionar el acceso y la asignación de asientos de una ceremonia de graduación universitaria (FIMAZ). El sistema permitirá a los alumnos confirmar sus invitados, generar un código QR temporal desde su casa, y al llegar al teatro, escanearlo en un kiosco. El sistema asignará asientos contiguos en tiempo real premiando la puntualidad, basándose en un mapa físico del teatro levantado con drones.

2. Stack Tecnológico

Lenguaje: PHP Vanilla (Puro, sin frameworks como Laravel o Symfony).

Base de Datos: MySQL (Usando PDO para la conexión y consultas preparadas).

Arquitectura: Patrón MVC+S (Modelo-Vista-Controlador-Servicio) adaptado para una API (No hay "Vistas" HTML en el backend, solo respuestas JSON).

3. Reglas de Arquitectura Estrictas (MVC+S)

Router (index.php): Es el único punto de entrada. Maneja CORS, evalúa el verbo HTTP (GET, POST, PUT, DELETE) y usa expresiones regulares con call_user_func_array para delegar la petición al Controlador correspondiente.

Controladores (Controllers): Son "delgados". Su única función es recibir los parámetros de la URL o el JSON del cuerpo (php://input), delegar la tarea al Servicio correspondiente y devolver un echo json_encode() con el código HTTP adecuado (200, 400, 401, 404, etc.). NUNCA se conectan a la base de datos ni llaman a un Modelo directamente.

Servicios (Services): Son los "cerebros". Contienen toda la lógica de negocio (validar reglas, algoritmos, criptografía). Son los únicos autorizados para llamar a los Modelos.

Modelos (Models): Son "tontos". Solo contienen consultas SQL (SELECT, INSERT, UPDATE). No toman decisiones lógicas.

4. Lógica de Negocio Principal

Gestión de Alumnos: El alumno confirma asistencia indicando el número de invitados (con un límite máximo).

Seguridad y QR: El sistema genera un Token (string encriptado, no una imagen) que contiene la cuenta del alumno y la marca de tiempo. Cuando el Kiosco lo escanea, el backend valida puramente con criptografía y tiempo si el QR fue generado hace menos de 5 minutos; si expiró, se rechaza.

Algoritmo de Asientos (Greedy): Al escanear un QR válido, el sistema busca en el mapa físico los N asientos libres contiguos más cercanos y los marca como ocupados.
